# City-Scale Infrastructure Benchmark

강남대로(도로 3구간) 케이스 스터디를 실제 "도시 규모"로 확장해, 지금까지
주장해온 "가볍다"(memory-light)는 claim을 실제 숫자로 검증한다. 두 개의
독립된 벤치마크로 나뉜다: 멀티에이전트 런타임(Registry+MetaAgent)과
Functorial Data Migration 레이어(checkInstanceIsFunctor/Δ/Σ/Π).

## 도메인 설계

```
RoadSegment (도로 구간)     — N까지 확장, 도시 규모의 진짜 변수
Intersection (교차로)       — 도로 5개당 1개 비율
AirQualityStation (대기질)  — 도로 20개당 1개 비율
```

세 개의 이질적 스키마를 실제 도시의 센서 밀도 비율(도로 촘촘, 대기질 센서
성김)로 섞었다 — 단순히 "같은 도메인을 N번 복제"가 아니라 진짜 이종 통합
부하 테스트가 되도록 설계했다.

## 실행

```bash
node --expose-gc -r ts-node/register city-infra/bench-agents.ts
node --expose-gc -r ts-node/register city-infra/bench-fdm.ts
```

`--expose-gc`는 측정 직전 강제 GC로 힙을 정리해 깨끗한 델타를 얻기 위함.
TypeScript 컴파일 옵션은 `TS_NODE_COMPILER_OPTIONS` 환경변수로 지정
(esModuleInterop 등, 기존 예제들과 동일).

## 결과 — 멀티에이전트 런타임 (Registry + MetaAgent)

| N(구간) | 총 도메인 | trimmed-mean 시간(ms) | 메모리 델타(MB) | 경고 수 |
|---:|---:|---:|---:|---:|
| 10 | 13 | 0.17 | 0.034 | 7 |
| 100 | 125 | 0.48 | 0.238 | 51 |
| 1,000 | 1,250 | 6.73 | 2.341 | 507 |
| 5,000 | 6,250 | 50.51 | 11.906 | 2,609 |

**메모리 스케일링 (도메인 수 대비):**
```
13 → 125 도메인:   N×9.6,  memory×7.1
125 → 1250 도메인: N×10.0, memory×9.8
1250 → 6250 도메인: N×5.0,  memory×5.1
```
거의 정확히 선형이다 — 6,250개 도메인(도로 구간 5,000개 규모, 서울시
자치구 하나 정도의 도로망에 해당하는 규모)을 전부 등록하고 한 번에
synthesis해도 메모리 증가분이 12MB 수준이다.

## 결과 — Functorial Data Migration 레이어

| N | checkInstanceIsFunctor(ms) | deltaF(ms) | sigmaF(ms) |
|---:|---:|---:|---:|
| 10 | 0.027 | 0.026 | 0.033 |
| 100 | 0.054 | 0.082 | 0.046 |
| 1,000 | 0.331 | 0.276 | 0.197 |
| 5,000 | 3.997 | 0.847 | 0.439 |

6,250행 규모에서도 전부 한 자리 밀리초 수준. `isFunctor: true`가 모든
N에서 확인됨(아래 "발견한 것" 참고).

**Π_F(조인)의 DoS 가드가 실제로 발동하는 지점:**
```
N=10   (10×2=20 행):       ✅ 허용
N=100  (100×20=2,000 행):  ✅ 허용
N=1000 (1000×200=200,000행): ⛔ 거부 (기본 cap 100,000 초과)
```
도로 구간과 교차로를 조인하면 도시 규모에서 실제로 20만 행 조합이
시도된다 — 지난 세션의 보안 패치(Π_F DoS 가드)가 장식이 아니라 이
규모에서 실제로 발동하는 방어선임을 확인했다.

## 정직하게 남길 것

**시간 측정은 메모리 측정보다 노이즈가 크다.** 같은 프로세스 안에서
반복 실행하면 힙이 누적되며 GC pause가 측정 구간에 불규칙하게
끼어든다 (N=1000 단독 30회 실행에서 3.3ms~27.3ms까지 분산 관찰). 이는
알고리즘 문제가 아니라 공유 컨테이너 환경의 벤치마킹 한계로 판단해
trimmed mean(느린 쪽 30% 제거)으로 완화했으나, 완전히 제거되진 않는다.
**따라서 "가볍다"는 주장의 핵심 증거는 시간이 아니라 메모리 델타다** —
이쪽은 첫 측정부터 도메인 수에 거의 정확히 비례했다.

**발견한 것 — checkInstanceIsFunctor가 우리 자신의 설계 실수를 잡아낸
사례.** 최초 버전의 격자 위상은 도시 "경계"를 표현하려고 각 행의
오른쪽 끝 도로에서 `connectsTo`를 비워뒀는데, Phase A 스키마에서는
`connectsTo`를 모든 RoadSegment에 대한 총함수로 선언해 놓은 상태였다.
그 결과 경계 노드들이 정확히 `not-total` 위반으로 잡혔다 — 데이터가
깨진 게 아니라 **스키마 선언과 실제 위상이 불일치**했던 것이다. 실제
도시 도로망을 정확히 모델링하려면 막다른 길/경계를 `connectsTo`의
부분함수(partial morphism)로 다루는 더 큰 설계가 필요하며, 이는 이번
벤치마크 범위 밖이라 순환 격자(toroidal grid, 경계 없음)로 단순화해
측정했다. 이 사례 자체는 Phase A 검증기가 실제로 유용하다는 실증
증거로 별도 가치가 있다고 판단해 기록해 둔다.

## 이 벤치마크가 증명하지 않는 것

- 절대 시간 수치를 프로덕션 성능 지표로 인용하지 말 것 — 공유 컨테이너
  환경이며, 전용 하드웨어에서의 벤치마크가 아니다. 여기서 의미 있는
  신호는 **스케일링 비율**(N이 늘 때 자원 사용이 어떻게 늘어나는가)이지
  절대값이 아니다.
- N=5,000(도로 구간)은 자치구 규모다. 서울시 전체 도로망(수만 개 링크)
  규모까지 이 선형성이 유지되는지는 별도 검증이 필요하다.
- Π_F(조인)는 애초에 수학적으로 조합적(combinatorial)이라 선형 스케일링
  대상이 아니다 — DoS 가드가 존재하는 이유가 바로 이것이며, 가드가
  "느려서 문제"가 아니라 "무한정 커지는 것 자체가 join의 본질"이라는
  점을 명확히 구분해야 한다.
